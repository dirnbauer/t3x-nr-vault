Vault Help

This guide explains how vault encryption works, how to manage secrets, and how to troubleshoot common issues.

Getting started

The Vault extension provides secure storage for sensitive configuration data (API keys, passwords, tokens) using envelope encryption. Follow these steps to get started:

Step 1 — Verify encryption health

Visit the Dashboard and check the health indicator at the top. A green banner confirms that encryption is active and working. If you see an error, ensure that $GLOBALS['TYPO3_CONF_VARS']['SYS']['encryptionKey'] is set in your TYPO3 configuration or configure an alternative master key provider.

Step 2 — Create your first secret

Navigate to Secrets and click the + button in the header. Choose a descriptive identifier (e.g., stripe_api_key) and enter the secret value. The value is encrypted before it reaches the database.

Step 3 — Reference secrets in configuration

Use the vault syntax anywhere TYPO3 resolves configuration values: 

%vault(stripe_api_key)%

This works in site configuration YAML, TypoScript, and any configuration that uses the TYPO3 expression language.

Step 4 — Migrate existing plaintext secrets (optional)

The Migration Wizard scans your database for plaintext secrets and guides you through encrypting them. It supports batch processing and verifies each migration before committing.

Envelope encryption

Vault uses a two-layer encryption scheme called envelope encryption, the same pattern used by AWS KMS, Google Cloud KMS, and Azure Key Vault:

How it works

  1. Data Encryption Key (DEK) — Each secret is encrypted with its own unique random key using AES-256-GCM. This means compromising one secret does not reveal others.
  2. Key Encryption Key (KEK / Master Key) — The DEK is then encrypted with the master key and stored alongside the ciphertext. The master key never leaves the server.

Master key providers

The master key can come from several sources (in priority order):

  • Environment variable (TYPO3_VAULT_MASTER_KEY) — recommended for production
  • File-based — a key file outside the web root
  • TYPO3 encryption key — fallback using $GLOBALS['TYPO3_CONF_VARS']['SYS']['encryptionKey']

Key rotation

The master key can be rotated without downtime using bin/typo3 vault:rotate-master-key. This re-encrypts all DEKs with the new master key. Individual secrets can also be rotated with bin/typo3 vault:rotate.

Audit trail

Every secret operation (create, read, update, delete, rotate) is recorded in a tamper-evident audit chain. Each entry includes:

  • The action performed and the secret identifier
  • The backend user who performed the action
  • A timestamp and IP address
  • A cryptographic hash linking each entry to the previous one

Use the Audit Log module to browse entries and the Verify Chain action to check integrity. Any tampered or missing entries will be flagged.

Troubleshooting

"Encryption is not available" on the dashboard

The master key provider could not be initialized. Check that at least one of the following is configured:

  • The TYPO3_VAULT_MASTER_KEY environment variable is set
  • A valid key file path is configured in extension settings
  • The TYPO3 encryptionKey is not empty

Secret values show as "decryption failed"

This means the master key has changed since the secret was encrypted. If the master key was rotated correctly, all DEKs should have been re-encrypted. If the key was lost, the affected secrets cannot be recovered — store a new value.

Migration wizard finds no candidates

The scanner looks for fields matching common secret patterns (columns named *password*, *key*, *token*, *secret*). If your field uses a different naming convention, you can add it manually via the CLI: bin/typo3 vault:migrate-field table_name field_name.

Audit chain verification fails

A broken chain means an entry was modified or deleted outside of vault operations. This could indicate tampering or a database restore from an inconsistent backup. The verification report shows exactly which entries are affected.
